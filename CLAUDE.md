# vexy-mkdocs-tags — agent notes

MkDocs plugin. Reads `tags:` from each page's YAML front matter and generates one
Markdown index grouping pages by tag.

## Layout
- `src/tags/plugin.py` — the plugin. `TagsPlugin(BasePlugin)` with `on_config`
  (resolve output path) and `on_files` (scan sources, write the tags file,
  append it to the build). `get_metadata()` parses the triple-dash YAML header.
- `src/tags/templates/tags.md.template` — default Markdown+Jinja2 template.
  Receives `tags`: a list of `(tag, pages)` pairs, sorted case-insensitively;
  pages sorted by title.
- `src/tags/_version.py` — generated by hatch-vcs, gitignored. Do not edit.
- `tests/test_plugin.py` — unit + edge cases. `tests/test_integration.py` —
  runs a real `mkdocs build` on a tiny fixture site.

## Entry point
Registered as `tags = "tags.plugin:TagsPlugin"` — the public plugin name is
`tags` (used in `mkdocs.yml`), so the top-level package stays `tags` even under
the `src/` layout.

## Conventions
- Versioning: hatch-vcs from `vX.Y.Z` git tags. Never hand-edit versions.
- `from __future__ import annotations` is required — annotations reference types
  lazily so imports stay light and Python 3.10 works.
- Verify: `pytest`, `ruff check src tests`, `ruff format --check src tests`, `mypy src`.
- Release: `uvx gitnextver@latest .` (commits, tags, pushes; release.yml publishes to PyPI).

## Watch out
- Default template output is the documented `# Contents grouped by tag` format.
  Earlier releases (≤0.3.2) shipped a divergent "Index of Topics" template that
  did not match the README; v1.0.0 aligns them.
- The tags file is written to `tags_folder` (default `aux`), resolved *beside*
  `docs_dir`, deliberately outside the docs tree to avoid serve rebuild loops.
